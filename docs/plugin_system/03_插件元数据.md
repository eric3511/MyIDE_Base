## 大纲

1. `MyPlugin.json.in` 是什么；
2. JSON 字段如何影响插件管理器；
3. `Q_PLUGIN_METADATA` 和 JSON 文件是什么关系；
4. `CMakeLists.txt` 中 `add_qtc_plugin` 做了什么；
5. `PLUGIN_DEPENDS` 和 JSON `Dependencies` 的区别；
6. 为什么 Qt Creator 可以不实例化插件对象就读取插件信息。

---

## 1. 插件元数据的作用

插件不是普通动态库。普通动态库只是一组可链接符号，而 Qt Creator 插件还必须告诉宿主程序：

- 我是谁；
- 我的版本是多少；
- 我依赖谁；
- 我是否默认启用；
- 我是否是实验性功能；
- 我能在哪些平台运行；
- 我加载失败时应该如何报告；
- 我是否和当前 Qt Creator 兼容。

这些信息就来自插件元数据。

---

### 1.1 插件元数据的位置

通常插件工程中有一个文件：


`MyPlugin.json.in`

构建后会生成：

`MyPlugin.json`

然后通过插件类中的：

```cpp
Q_PLUGIN_METADATA(IID "org.qt-project.Qt.QtCreatorPlugin" FILE "MyPlugin.json")
```

嵌入到插件动态库中。

整体关系是：

```text
MyPlugin.json.in
   ↓ CMake 配置 / 变量替换
MyPlugin.json
   ↓ Q_PLUGIN_METADATA
嵌入插件动态库
   ↓ QPluginLoader::metaData()
Qt Creator 读取元数据
   ↓
生成 PluginSpec
```


---

### 1.2 为什么是 `.json.in`

`.json.in` 里的 `.in` 表示它是模板文件。

因为有些字段可能不是手写死值，而是由 CMake 替换，例如：

```json
{
    "Version" : "%{QtCreatorPluginVersion}",
    "CompatVersion" : "%{QtCreatorPluginCompatVersion}"
}
```

或者某些版本使用类似：

```json
{
    "Version" : "@QTCREATOR_VERSION@",
    "CompatVersion" : "@QTCREATOR_COMPAT_VERSION@"
}
```


具体占位符形式以当前 Qt Creator 版本为准。

核心思想是：

> `.json.in` 是源码模板，最终真正嵌入动态库的是构建生成后的 `.json`。

---

## 2. JSON 元数据字段详解

一个典型插件元数据如下：

```json
{
    "Name" : "MyPlugin",
    "Version" : "1.0.0",
    "CompatVersion" : "1.0.0",
    "Vendor" : "Me",
    "Copyright" : "(C) 2026 Me",
    "License" : "Commercial",
    "Category" : "Examples",
    "Description" : "A minimal Qt Creator plugin example.",
    "Url" : "https://example.com",
    "Dependencies" : [
        { "Name" : "Core", "Version" : "1.0.0" }
    ]
}
```


这些字段不是摆设。它们会直接影响插件能否被识别、显示、加载和排序。

---

### 2.1 核心字段说明

|**字段**|**作用**|**是否关键**|
|---|---|---|
|`Name`|插件唯一名称|是|
|`Version`|插件自身版本|是|
|`CompatVersion`|插件兼容版本|是|
|`Vendor`|插件提供者|否，但建议填写|
|`Category`|插件管理器中的分类|否，但建议填写|
|`Description`|插件描述|否，但建议填写|
|`Url`|插件相关链接|否|
|`Dependencies`|插件运行时依赖|是|
|`DisabledByDefault`|是否默认禁用|可选|
|`Experimental`|是否实验性插件|可选|
|`Platform`|平台限制|可选|

---

### 2.2 `Name`


`"Name" : "MyPlugin"`

`Name` 是插件唯一标识。

它用于：

- 插件列表展示；
- 依赖解析；
- 插件启用 / 禁用配置；
- 错误信息定位；
- 其他插件声明依赖。

注意：


`"Name" : "MyPlugin"`

和 C++ 类名：

`MyPluginPlugin`

不是一回事。

推荐保持命名清晰一致，例如：

|**概念**|**示例**|
|---|---|
|插件元数据名称|`MyPlugin`|
|插件 C++ 类名|`MyPluginPlugin`|
|插件 CMake target|`MyPlugin`|
|Action ID 前缀|`MyPlugin.Action`|

---

### 2.3 `Version`


`"Version" : "1.0.0"`

表示插件当前版本。

它用于：

- 插件管理器显示；
- 依赖版本判断；
- 兼容性检查；
- 错误信息提示。

版本通常采用类似语义化版本：


`major.minor.patch`

例如：

```txt
1.0.0
12.0.2
14.1.0
```

---

### 2.4 `CompatVersion`


`"CompatVersion" : "1.0.0"`

表示兼容版本。

它的意义是：

> 当前插件版本可以兼容从某个版本开始的依赖关系。

例如某插件：

```json
"Version" : "2.3.0",
"CompatVersion" : "2.0.0"
```


可以理解为：

`这个插件当前是 2.3.0，但对依赖方来说，2.0.0 以上的接口仍然兼容。`

这对插件依赖版本判断很重要。

---

### 2.5 `Dependencies`

```json
"Dependencies" : [
    { "Name" : "Core", "Version" : "1.0.0" }
]
```


这是非常关键的字段。

它告诉 Qt Creator：

> 我的插件运行前，需要这些插件先准备好。

例如你的插件使用了：

```cpp
Core::ActionManager
Core::ICore
Core::Constants
```

那么你就应该声明依赖：

```json
"Dependencies" : [
    { "Name" : "Core", "Version" : "..." }
]
```


否则可能出现：

- 编译能过；
- 运行时加载顺序不对；
- `Core` 还没初始化，你就使用它；
- 插件行为异常；
- 加载失败。

---

### 2.6 可选字段

#### `DisabledByDefault`

```json
"DisabledByDefault" : true
```


表示插件默认禁用。

适合：

- 实验性插件；
- 不稳定插件；
- 需要用户手动开启的功能；
- 特定场景才需要的插件。

---

#### `Experimental`

`"Experimental" : true`

表示实验性插件。

插件管理器通常会以特殊方式展示，让用户知道这是不稳定功能。

---

#### `Platform`



`"Platform" : "windows"`

或者某些版本中支持更复杂的表达方式。

它用于限制插件运行平台。

适合：

- 只支持 Windows 的调试器；
- 只支持 Linux 的工具链集成；
- 依赖特定平台 SDK 的插件。

---

## 3. `Q_PLUGIN_METADATA` 与 JSON 的关系

在插件类中通常有：

```cpp
Q_PLUGIN_METADATA(IID "org.qt-project.Qt.QtCreatorPlugin" FILE "MyPlugin.json")
```


这行宏非常关键。

---

### 3.1 它做了什么

它做两件事：

1. 标记这个类是 Qt 插件；
2. 把 `MyPlugin.json` 作为插件元数据嵌入动态库。

简化理解：

```text
C++ 插件类
   +
Q_PLUGIN_METADATA
   +
MyPlugin.json
   ↓
生成一个可被 QPluginLoader 识别的 Qt 插件动态库
```


---

### 3.2 `IID` 是什么


`IID "org.qt-project.Qt.QtCreatorPlugin"`

`IID` 是接口标识。

它告诉 Qt 插件系统：

`这个插件实现的是 Qt Creator 插件接口。`

如果 `IID` 错误，Qt Creator 可能不会把它当成合法插件。

---

### 3.3 `FILE` 必须匹配生成文件

`FILE "MyPlugin.json"`

这里写的是最终生成的 JSON 文件名，不一定是 `.json.in`。

所以要区分：

```text
源码模板：MyPlugin.json.in
构建生成：MyPlugin.json
宏中引用：MyPlugin.json
```


如果文件名不匹配，常见结果是：

- 编译时报错；
- 插件元数据为空；
- 插件无法被识别；
- Qt Creator 插件管理器中看不到它。

---

## 4. Qt Creator 如何读取插件元数据

这是本课的关键理解点。

Qt Creator 不需要先创建插件对象，就可以读取插件元数据。

因为 Qt 提供了：`QPluginLoader::metaData()`

简化流程如下：

```text
找到插件动态库
   ↓
创建 QPluginLoader
   ↓
调用 metaData()
   ↓
读取嵌入的 JSON
   ↓
创建 PluginSpec
   ↓
检查依赖和兼容性
   ↓
决定是否真正加载插件
```

---

### 4.1 为什么不先实例化插件

如果 Qt Creator 一扫描到动态库就直接创建插件对象，会有很多问题：

- 依赖可能还没准备好；
- 插件可能平台不兼容；
- 插件可能被用户禁用；
- 插件可能版本不兼容；
- 插件构造函数里可能执行危险逻辑；
- 错误处理会很混乱。

所以正确流程是：
`先读元数据 -> 再判断能不能加载 -> 最后才创建插件对象`

这就是 `PluginSpec` 存在的意义之一。

---

### 4.2 对应到 `PluginSpec`

元数据最终会进入 `PluginSpec`。

可以这样理解：

```text
QPluginLoader::metaData()
   ↓
JSON Object
   ↓
PluginSpec::read(...)
   ↓
PluginSpec 内部字段
```


大致对应关系：

|**JSON 字段**|**PluginSpec 中的概念**|
|---|---|
|`Name`|插件名称|
|`Version`|插件版本|
|`CompatVersion`|兼容版本|
|`Dependencies`|依赖列表|
|`Description`|描述|
|`Vendor`|提供者|
|`Category`|分类|
|加载错误|错误状态 / 错误信息|
|插件启用状态|enabled / disabled 相关字段|

## 5. CMake 插件构建基础

前面讲的是“插件如何被描述”，现在讲“插件如何被构建”。

在 Qt Creator 源码树内，一个插件的 CMake 通常类似：

```cmake
add_qtc_plugin(MyPlugin
  PLUGIN_DEPENDS
    Core
  SOURCES
    myplugin.h
    myplugin.cpp
    MyPlugin.json.in
)
```


不同 Qt Creator 版本参数可能略有差异，但核心思想一致。

---

### 5.1 `add_qtc_plugin` 的作用

你可以先把它理解为 Qt Creator 封装好的插件构建宏。

它大致负责：

- 创建插件动态库 target；
- 设置输出目录；
- 处理 `.json.in`；
- 生成最终 JSON；
- 调用 Qt MOC；
- 链接 Qt Creator 内部库；
- 链接依赖插件；
- 设置插件导出宏；
- 让插件能被 Qt Creator 扫描到。

---

### 5.2 为什么不用普通 `add_library`

理论上插件也是动态库，你可能会想：

`add_library(MyPlugin SHARED ...)`

但 Qt Creator 插件还需要处理很多额外事情：

- 插件输出位置；
- 插件元数据；
- Qt 插件宏；
- Qt Creator 插件依赖；
- 平台差异；
- Debug / Release 后缀；
- 安装路径；
- RPATH；
- 导出符号；
- 内部库链接。

所以使用：

`add_qtc_plugin(...)`

更安全。

普通 `add_library` 不是不能做，而是你要手动补齐很多细节，容易快乐地掉坑。

---

### 5.3 `PLUGIN_DEPENDS`

```text
PLUGIN_DEPENDS
  Core
```

表示编译和链接层面的依赖。

因为你的代码用了：

```cpp
#include <coreplugin/actionmanager/actionmanager.h>
#include <coreplugin/icore.h>
```


并调用了：

```cpp
Core::ActionManager
Core::ICore
```


所以编译时和链接时需要 `Core`。

---

## 6. 编译时依赖 vs 运行时依赖

这是本课非常重要的概念。

你的插件依赖 `Core`，通常需要在两个地方声明：

### 6.1 CMake 中声明

```cmake
PLUGIN_DEPENDS
  Core
```


作用是：


`编译 / 链接时知道我要用 Core 的头文件、库、target`

---

### 6.2 JSON 中声明

```json
"Dependencies" : [
    { "Name" : "Core", "Version" : "..." }
]
```


作用是：
`运行时 PluginManager 知道我要等 Core 准备好之后再加载`

---

### 6.3 二者区别

|**位置**|**依赖类型**|**解决的问题**|
|---|---|---|
|`CMakeLists.txt`|编译时依赖|代码能编译、能链接|
|`MyPlugin.json.in`|运行时依赖|插件能按正确顺序加载|
|`#include`|源码级依赖|代码能看到类型声明|
|对象池查询|运行时服务依赖|插件能动态发现服务|

一句话总结：

> CMake 解决“能不能编译”，JSON 解决“能不能安全加载”。

---

### 6.4 常见错误

#### 只写 CMake，不写 JSON

`PLUGIN_DEPENDS   Core`

但 JSON 没有：

`"Dependencies" : [     { "Name" : "Core", "Version" : "..." } ]`

可能导致：

- 编译成功；
- 运行时加载顺序不受保障；
- 插件依赖关系不完整；
- 插件管理器无法准确展示依赖。

---

#### 只写 JSON，不写 CMake

JSON 有：

`"Dependencies" : [     { "Name" : "Core", "Version" : "..." } ]`

但 CMake 没有：

`PLUGIN_DEPENDS   Core`

可能导致：

- 找不到头文件；
- 链接失败；
- 未解析外部符号；
- 构建系统不知道依赖关系。

---

## 7. 插件版本兼容理解

版本字段后面读 `PluginSpec` 时会详细研究，现在先建立基本模型。

---

### 7.1 依赖版本示例

假设 `TextEditor` 插件元数据是：

```json
{
    "Name" : "TextEditor",
    "Version" : "5.2.0",
    "CompatVersion" : "5.0.0"
}
```


你的插件声明：

```json
"Dependencies" : [
    { "Name" : "TextEditor", "Version" : "5.1.0" }
]
```


大致可以理解为：


`我需要 TextEditor 至少具备 5.1.0 级别的兼容能力。`

如果 `TextEditor` 当前版本是 `5.2.0`，兼容到 `5.0.0`，通常可以满足。

---

### 7.2 版本不兼容的后果

如果版本不兼容，可能发生：

- 当前插件不会被加载；
- `PluginSpec` 标记错误；
- 插件管理 UI 显示依赖失败；
- 依赖当前插件的其他插件也可能被跳过。

这就是为什么插件版本和兼容版本不能随便写。
