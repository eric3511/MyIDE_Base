## 最小插件闭环

这一课我们做一个最小插件，功能非常简单：

1. Qt Creator 启动时打印日志；
2. 插件初始化时注册一个菜单 Action；
3. 点击 Action 后弹出消息框；
4. Qt Creator 关闭时打印日志并清理资源。

完成后，你应该能回答：

- 插件类是如何声明的？
- `Q_PLUGIN_METADATA` 有什么作用？
- `initialize()` 什么时候执行？
- 菜单 Action 一般在哪里注册？
- `aboutToShutdown()` 什么时候触发？

---

## 1. 最小插件的整体结构

一个 Qt Creator C++ 插件通常至少包含这些文件：

```shell
myplugin/
   |
   +-- myplugin.h
   +-- myplugin.cpp
   +-- myplugin.json.in
   +-- CMakeLists.txt
```


如果使用 Qt Creator 自带插件向导，可能还会生成：

```shell
myplugin_global.h
constants.h
```

不过最核心的仍然是这四个：

|**文件**|**作用**|
|---|---|
|`myplugin.h`|声明插件类，继承 `ExtensionSystem::IPlugin`|
|`myplugin.cpp`|实现生命周期函数和插件逻辑|
|`myplugin.json.in`|插件元数据模板|
|`CMakeLists.txt`|构建插件动态库|

---

## 2. 插件类声明

插件类通常继承：

`ExtensionSystem::IPlugin`

一个最小插件头文件可以长这样：

```cpp
#pragma once

#include <extensionsystem/iplugin.h>

namespace MyPlugin {
namespace Internal {

class MyPluginPlugin final : public ExtensionSystem::IPlugin
{
    Q_OBJECT
    Q_PLUGIN_METADATA(IID "org.qt-project.Qt.QtCreatorPlugin" FILE "MyPlugin.json")

public:
    MyPluginPlugin();
    ~MyPluginPlugin() override;

    bool initialize(const QStringList &arguments, QString *errorString) override;
    void extensionsInitialized() override;
    ShutdownFlag aboutToShutdown() override;
};

} // namespace Internal
} // namespace MyPlugin
```

---

### 2.1 关键点解释

这里有几个点非常重要。

#### `Q_OBJECT`

这是 Qt 元对象系统需要的宏。

它支持：

- 信号槽；
- 元对象信息；
- `qobject_cast`；
- 插件系统相关元信息。

---

#### `Q_PLUGIN_METADATA`

这一行很关键：

```cpp
Q_PLUGIN_METADATA(IID "org.qt-project.Qt.QtCreatorPlugin" FILE "MyPlugin.json")
```

它告诉 Qt：

- 这是一个 Qt 插件；
- 插件接口 ID 是 `org.qt-project.Qt.QtCreatorPlugin`；
- 插件元数据来自 `MyPlugin.json`。

Qt Creator 后面可以通过 `QPluginLoader::metaData()` 读取这些信息。

也就是说：

```text
Q_PLUGIN_METADATA
   ↓
把 JSON 元数据嵌入插件动态库
   ↓
Qt Creator 扫描动态库时读取元数据
   ↓
生成 PluginSpec
```


---

#### `initialize()`

```cpp
bool initialize(const QStringList &arguments, QString *errorString) override;
```

插件初始化入口。

返回：

- `true`：初始化成功；
- `false`：初始化失败，插件不可用。

如果失败，可以设置：

`*errorString = "Failed to initialize MyPlugin";`

---

#### `extensionsInitialized()`

```cpp
void extensionsInitialized() override;
```

所有插件 `initialize()` 完成后调用。

这时适合：

- 获取其他插件注册的服务；
- 做跨插件连接；
- 启动依赖全局环境的逻辑。

---

#### `aboutToShutdown()`

```cpp
ShutdownFlag aboutToShutdown() override;
```

Qt Creator 关闭前调用。

常用于：

- 停止后台线程；
- 保存状态；
- 移除对象池对象；
- 断开连接；
- 释放资源。

---

## 3. 插件实现文件

一个最小实现可以这样写。

```cpp
#include "myplugin.h"

#include <coreplugin/actionmanager/actionmanager.h>
#include <coreplugin/actionmanager/command.h>
#include <coreplugin/coreconstants.h>
#include <coreplugin/icore.h>

#include <QAction>
#include <QDebug>
#include <QMessageBox>

using namespace Core;

namespace MyPlugin {
namespace Internal {

MyPluginPlugin::MyPluginPlugin()
{
    qDebug() << "MyPlugin constructor";
}

MyPluginPlugin::~MyPluginPlugin()
{
    qDebug() << "MyPlugin destructor";
}

bool MyPluginPlugin::initialize(const QStringList &arguments, QString *errorString)
{
    Q_UNUSED(arguments)
    Q_UNUSED(errorString)

    qDebug() << "MyPlugin initialize";

    auto action = new QAction(tr("My Plugin Action"), this);

    Command *cmd = ActionManager::registerAction(
        action,
        "MyPlugin.Action",
        Context(Constants::C_GLOBAL)
    );

    Q_UNUSED(cmd)

    ActionContainer *menu = ActionManager::actionContainer(Constants::M_TOOLS);
    menu->addAction(cmd);

    connect(action, &QAction::triggered, this, [] {
        QMessageBox::information(
            Core::ICore::dialogParent(),
            QObject::tr("MyPlugin"),
            QObject::tr("Hello from MyPlugin")
        );
    });

    return true;
}

void MyPluginPlugin::extensionsInitialized()
{
    qDebug() << "MyPlugin extensionsInitialized";
}

ExtensionSystem::IPlugin::ShutdownFlag MyPluginPlugin::aboutToShutdown()
{
    qDebug() << "MyPlugin aboutToShutdown";
    return SynchronousShutdown;
}

} // namespace Internal
} // namespace MyPlugin
```

---

## 4. 代码逐段解释

这一节把代码拆开讲清楚。你现在不用追 `ActionManager` 的内部实现，只需要理解它是 Core 插件提供的菜单和 Action 管理能力。

---

### 4.1 构造和析构

```cpp
MyPluginPlugin::MyPluginPlugin()
{
    qDebug() << "MyPlugin constructor";
}

MyPluginPlugin::~MyPluginPlugin()
{
    qDebug() << "MyPlugin destructor";
}
```

这里用于观察插件对象什么时候被创建和销毁。

你后面启动 Qt Creator 时，应该能看到类似日志：

```shell
MyPlugin constructor
MyPlugin initialize
MyPlugin extensionsInitialized
...
MyPlugin aboutToShutdown
MyPlugin destructor
```

注意：构造函数不适合做复杂初始化。

推荐把主要逻辑放在：

```cpp
initialize()
extensionsInitialized()
delayedInitialize()
```

---

### 4.2 `initialize()`

```cpp
bool MyPluginPlugin::initialize(const QStringList &arguments, QString *errorString)
```


这里是插件初始化主入口。

本例中做三件事：

1. 打印日志；
2. 创建 Action；
3. 添加到 Tools 菜单；
4. 绑定点击事件。

`Q_UNUSED(arguments)`
`Q_UNUSED(errorString)`

表示暂时不用命令行参数和错误字符串。

---

### 4.3 创建 QAction

```cpp
auto action = new QAction(tr("My Plugin Action"), this);
```

这里创建一个菜单动作。

注意父对象是 `this`，也就是插件对象。

这样做的好处是：

- 插件对象销毁时，Action 也会跟着销毁；
- 避免手动 delete；
- 避免内存泄漏。

---

### 4.4 注册 Action

```cpp
Command *cmd = ActionManager::registerAction(
    action,
    "MyPlugin.Action",
    Context(Constants::C_GLOBAL)
);
```

这里把 `QAction` 注册到 Qt Creator 的 Action 系统中。

参数含义：

|**参数**|**含义**|
|---|---|
|`action`|要注册的 QAction|
|`"MyPlugin.Action"`|Action 的唯一 ID|
|`Context(Constants::C_GLOBAL)`|全局上下文|

`ActionManager` 来自 `Core` 插件。

这说明你的插件通常需要依赖 `Core` 插件，否则你无法使用菜单系统、主窗口、对话框父窗口等核心 UI 能力。

---

### 4.5 添加到 Tools 菜单

```cpp
ActionContainer *menu = ActionManager::actionContainer(Constants::M_TOOLS);
menu->addAction(cmd);
```


这段代码的意思是：

```text
找到 Qt Creator 的 Tools 菜单
   ↓
把我的 Action 加进去
```

启动后，你应该能在菜单栏看到：

`Tools > My Plugin Action`

---

### 4.6 点击弹窗

```cpp
connect(action, &QAction::triggered, this, [] {
    QMessageBox::information(
        Core::ICore::dialogParent(),
        QObject::tr("MyPlugin"),
        QObject::tr("Hello from MyPlugin")
    );
});
```


点击菜单项后弹出消息框。

这里使用：

`Core::ICore::dialogParent()`

作为父窗口。

这样弹窗会正确归属于 Qt Creator 主窗口，而不是变成一个孤零零的小窗口。

---

### 4.7 `extensionsInitialized()`

```cpp
void MyPluginPlugin::extensionsInitialized()
{
    qDebug() << "MyPlugin extensionsInitialized";
}
```


本课只打印日志。

后面你会把跨插件连接放到这里。

例如：

- 从对象池获取 IProjectManager
- 从对象池获取 IEditorFactory
- 连接服务之间的信号槽


---

### 4.8 `aboutToShutdown()`

```cpp
ExtensionSystem::IPlugin::ShutdownFlag MyPluginPlugin::aboutToShutdown()
{
    qDebug() << "MyPlugin aboutToShutdown";
    return SynchronousShutdown;
}
```


这里返回：

`SynchronousShutdown`

表示可以同步关闭。

如果插件有后台任务，可能需要异步关闭，后面高级课再讲。

## 5. 插件 JSON 元数据

插件通常有一个：

`MyPlugin.json.in`

示例：

```json
{
    "Name" : "MyPlugin",
    "Version" : "1.0.0",
    "CompatVersion" : "1.0.0",
    "Vendor" : "Me",
    "Copyright" : "(C) 2026 Me",
    "License" : "Commercial",
    "Category" : "Examples",
    "Description" : "A minimal Qt Creator plugin example.",
    "Url" : "https://example.com",
    "Dependencies" : [
        { "Name" : "Core", "Version" : "1.0.0" }
    ]
}
```
---

### 5.1 字段说明

|**字段**|**说明**|
|---|---|
|`Name`|插件唯一名称|
|`Version`|插件版本|
|`CompatVersion`|兼容版本|
|`Vendor`|插件提供者|
|`Category`|插件分类|
|`Description`|插件描述|
|`Url`|插件相关链接|
|`Dependencies`|插件依赖|

本插件依赖：

`{ "Name" : "Core", "Version" : "1.0.0" }`

因为我们使用了：

`Core::ActionManager`
`Core::ICore`
`Core::Constants`

---

### 5.2 关于版本号的提醒

真实 Qt Creator 插件的依赖版本需要和当前 Qt Creator 的插件版本匹配。

示例中的：


`"Version" : "1.0.0"`

只是教学示例。

实际项目中，要参考当前 Qt Creator 源码或生成模板里的版本变量。

很多官方模板会使用类似占位符或 CMake 变量生成最终 JSON。

## 6. CMakeLists.txt 基础结构

如果是在 Qt Creator 源码树内开发插件，CMake 可能类似：

```cmake
add_qtc_plugin(MyPlugin
  PLUGIN_DEPENDS
    Core
  SOURCES
    myplugin.h
    myplugin.cpp
    MyPlugin.json.in
)
```

---

### 6.1 `add_qtc_plugin` 的作用

你现在可以先把它理解成 Qt Creator 提供的插件构建宏。

它大致负责：

- 生成插件动态库；
- 处理插件 JSON 元数据；
- 设置插件输出目录；
- 链接依赖插件；
- 配置 Qt MOC；
- 设置导出符号；
- 让 Qt Creator 能找到这个插件。

---

### 6.2 `PLUGIN_DEPENDS`

```text
PLUGIN_DEPENDS
  Core
```

表示构建层面依赖 `Core` 插件。

注意这里有两个层面的依赖：

|**依赖位置**|**作用**|
|---|---|
|`CMakeLists.txt` 中的 `PLUGIN_DEPENDS Core`|编译和链接时需要 Core|
|`MyPlugin.json.in` 中的 `Dependencies`|运行时插件加载顺序和依赖检查|

两者都重要。
