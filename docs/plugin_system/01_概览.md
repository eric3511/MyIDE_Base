## 概览
Plugin 是 Qt Creator 中最强的扩展方式，因为它可以进入 Qt Creator 的核心运行时环境。

它可以做的事情包括：
- 添加菜单、工具栏、快捷键；
- 添加 Dock 面板；
- 添加新的编辑器；
- 添加新的项目类型；
- 集成编译器；
- 集成调试器；
- 增加代码分析、补全、诊断；
- 扩展设置页面；
- 和其他插件共享服务对象。

## 1.Qt Creator 的微内核架构
Qt Creator 的核心思想可以用一句话概括：

> 主程序只保留最基础的启动和插件加载能力，大部分 IDE 功能都由插件提供。

这就是典型的 微内核架构。

### 什么是微内核架构

微内核架构的特点是：

- 核心非常小；
- 业务能力通过插件扩展；
- 插件可以独立加载；
- 插件之间通过接口通信；
- 插件生命周期由统一管理器调度。

可以把它类比成一个“插排”：

```text
PluginManager = 插排
Plugin        = 插头设备
Object Pool   = 公共服务区
```


主程序不需要知道每个插件内部怎么实现，只负责：

- 找到插件；
- 读取插件信息；
- 判断能不能加载；
- 按顺序加载；
- 调用生命周期函数；
- 管理插件暴露出来的对象。

## 2.Qt Creator 的结构抽象
从架构上看，Qt Creator 大致可以理解成：
```text
Qt Creator Executable
   |
   +-- ExtensionSystem
   |      |
   |      +-- IPlugin
   |      +-- PluginSpec
   |      +-- PluginManager
   |      +-- Object Pool
   |
   +-- CorePlugin
   +-- TextEditorPlugin
   +-- ProjectExplorerPlugin
   +-- DebuggerPlugin
   +-- CppEditorPlugin
   +-- GitPlugin
   +-- YourPlugin
```
其中：

| **组件**                  | **作用**                   |
| ----------------------- | ------------------------ |
| `Qt Creator Executable` | 启动程序，创建应用对象，初始化插件系统      |
| `ExtensionSystem`       | 插件运行时框架                  |
| `IPlugin`               | 所有插件的基类                  |
| `PluginSpec`            | 插件描述对象，保存元数据、状态、依赖、错误    |
| `PluginManager`         | 插件系统总调度器                 |
| `Object Pool`           | 插件间共享对象池                 |
| `CorePlugin`            | Qt Creator 最基础的 UI 和服务插件 |
| `TextEditorPlugin`      | 文本编辑能力                   |
| `ProjectExplorerPlugin` | 工程管理能力                   |
| `YourPlugin`            | 你自己写的插件                  |

---

### 为什么不做成单体架构

如果 Qt Creator 是单体架构，可能会变成这样：

```text
MainWindow
   |
   +-- C++ 编辑器代码
   +-- Git 代码
   +-- Debugger 代码
   +-- Project 代码
   +-- Analyzer 代码
   +-- Terminal 代码
   +-- Python 支持代码
   +-- QML 支持代码
   +-- 所有其他功能
```


这会导致：

- 编译慢；
- 模块耦合严重；
- 功能难以裁剪；
- 新功能难以独立开发；
- 某个模块出错可能影响全局；
- 第三方扩展困难。

插件架构则变成：

```text
Main Application
   |
   +-- PluginManager
          |
          +-- CorePlugin
          +-- TextEditorPlugin
          +-- GitPlugin
          +-- DebuggerPlugin
          +-- YourPlugin
```


优势是：

- 模块边界清晰；
- 功能可以按需加载；
- 依赖关系明确；
- 扩展更容易；
- 插件可以独立演进；
- 更适合大型 IDE。

## 3. 插件系统的四个核心概念
```text
IPlugin
PluginSpec
PluginManager
Object Pool
```

先建立直觉即可，不需要现在深挖源码。

### 3.1 IPlugin：插件基类
IPlugin 是所有 Qt Creator C++ 插件的基类。

你的插件大概会长这样：

```cpp
class MyPlugin final : public ExtensionSystem::IPlugin
{
    Q_OBJECT
    Q_PLUGIN_METADATA(IID "org.qt-project.Qt.QtCreatorPlugin" FILE "MyPlugin.json")

public:
    bool initialize(const QStringList &arguments, QString *errorString) override;
    void extensionsInitialized() override;
    ShutdownFlag aboutToShutdown() override;
};
```

它定义了插件生命周期：

```text
initialize()
   ↓
extensionsInitialized()
   ↓
delayedInitialize()
   ↓
aboutToShutdown()
```

现在只需要记住：

    initialize():插件开始初始化自己
    extensionsInitialized():所有插件基本初始化完成，可以做跨插件连接
    delayedInitialize():延迟做不紧急或耗时的事情
    aboutToShutdown():Qt Creator 关闭前释放资源

### 3.2 PluginSpec：插件说明书
PluginSpec 可以理解成插件的“档案袋”。

它记录：

- 插件名字；
- 插件版本；
- 插件路径；
- 插件依赖；
- 插件状态；
- 插件是否启用；
- 插件错误信息；
- 插件对应的实际对象。
比如插件 JSON 中有：

```json
{
    "Name" : "MyPlugin",
    "Version" : "1.0.0",
    "CompatVersion" : "1.0.0",
    "Vendor" : "Me",
    "Description" : "Example plugin",
    "Dependencies" : [
        { "Name" : "Core", "Version" : "1.0.0" }
    ]
}
```

这些信息会被读取并形成一个运行时描述对象，也就是 PluginSpec 的职责范围。

简单理解：

```text
JSON 元数据 + 动态库路径 + 状态 + 错误信息 = PluginSpec
```

### 3.3 PluginManager：插件总调度器
PluginManager 是最核心的类。

它负责：

- 设置插件搜索路径；
- 扫描插件动态库；
- 读取插件元数据；
- 创建 PluginSpec；
- 解析依赖；
- 决定加载顺序；
- 加载插件；
- 创建插件实例；
- 调用生命周期函数；
- 管理对象池；
- 处理错误；
- 关闭插件。

可以把它理解为：
**PluginManager = 插件系统的操作系统**
一个简化流程如下：
```text
setPluginPaths()
   ↓
scan plugins
   ↓
read metadata
   ↓
create PluginSpec
   ↓
resolve dependencies
   ↓
load plugin libraries
   ↓
create IPlugin instances
   ↓
call initialize()
   ↓
call extensionsInitialized()
   ↓
call delayedInitialize()
   ↓
run
   ↓
aboutToShutdown()
```

### 3.4 Object Pool：插件间通信的公共服务池
对象池是 Qt Creator 插件系统非常关键的设计。

插件之间不建议这样：

```cpp
auto editor = new TextEditorInternalImplementation;
```

也不应该到处直接依赖具体实现类。

更推荐这样：

**TextEditorPlugin 注册一个编辑器服务对象**
**YourPlugin 从 PluginManager 对象池中按接口查询它**
概念上：

```cpp
PluginManager::addObject(object);
PluginManager::getObject<T>();
PluginManager::getObjects<T>();
PluginManager::removeObject(object);
```

它解决的问题是：

- 插件之间低耦合；
- 插件不依赖具体实现；
- 支持多个同类扩展；
- 支持运行时发现服务；
- 支持软依赖。

## 4. 插件生命周期初步理解

先建立生命周期概念，后面会结合源码细读。

---

### 4.1 生命周期顺序

一个插件大致经历：

```text
发现插件动态库
   ↓
读取 JSON 元数据
   ↓
解析依赖
   ↓
加载动态库
   ↓
创建插件对象
   ↓
initialize()
   ↓
extensionsInitialized()
   ↓
delayedInitialize()
   ↓
运行中
   ↓
aboutToShutdown()
   ↓
销毁插件对象
```

注意这里有一个关键点：

> 读取插件元数据不等于真正加载并运行插件。

Qt Creator 可以先读取插件 JSON，判断依赖是否满足，再决定是否真正创建插件实例。

---

### 4.2 `initialize()` 的含义

`initialize()` 是插件自己的初始化阶段。

适合做：

- 创建核心对象；
- 注册服务对象；
- 添加菜单；
- 添加 Action；
- 读取配置；
- 检查命令行参数；
- 注册设置页。

不适合做：

- 大量耗时任务；
- 假设所有插件都已经完全准备好；
- 复杂跨插件业务联动。

---

### 4.3 `extensionsInitialized()` 的含义

`extensionsInitialized()` 会在所有插件的 `initialize()` 调用结束后执行。

适合做：

- 查询其他插件注册的对象；
- 建立跨插件连接；
- 做依赖全局环境的初始化；
- 连接服务之间的信号槽。

例如：

```text
ProjectExplorer 已经注册项目服务
TextEditor 已经注册编辑器服务
你的插件现在可以把两者连接起来
```

---

### 4.4 `delayedInitialize()` 的含义

`delayedInitialize()` 用于延迟初始化。

适合做：

- 后台索引；
- 启动语言服务；
- 扫描设备；
- 加载大型缓存；
- 初始化非首屏必要功能。

对于 IDE 来说，尤其适合：

- 扫描 PLC 设备
- 连接远程控制器
- 加载历史变量数据
- 启动语法检查进程
- 加载在线监控后台服务

这些事情不应该阻塞 IDE 启动。

---

### 4.5 `aboutToShutdown()` 的含义

`aboutToShutdown()` 在关闭前调用。

适合做：

- 停止后台线程；
- 断开设备连接；
- 保存状态；
- 移除对象池对象；
- 关闭文件；
- 释放外部资源。

对于IDE 来说，尤其要注意：

```text
关闭前必须安全断开 PLC 通信
停止在线监控
保存工程状态
停止编译后台进程
```


否则很容易出现“IDE 关了，后台线程还在快乐奔跑”的小型灾难现场。
